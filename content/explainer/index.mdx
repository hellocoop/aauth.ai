# AAuth Explainer

AAuth is an authentication and authorization protocol for autonomous agents and headless systems. It is **not** intended as a replacement for OAuth or OIDC — it is not OAuth 3.0. It is a complementary protocol for use cases where OAuth's assumptions — pre-registered clients, browser redirects, bearer tokens, static scopes — do not apply.

## HTTP Clients Need Their Own Identity

In OAuth 2.0 and OpenID Connect, the client has no independent identity. A `client_id` at Google is meaningless at GitHub. The client's identity exists only in the context of each server it has pre-registered with. API keys push the same model further: a shared secret that must be copied to where the workload runs — and will eventually be copied somewhere it shouldn't be.

SPIFFE and WIMSE brought workload identity to enterprise infrastructure, but these operate within a single trust domain. They don't help an agent that needs to access resources across organizational boundaries.

AAuth starts from this premise: **every agent has its own cryptographic identity.** An agent identifier (`aauth:local@domain`) is bound to a signing key, published at a well-known URL, and verifiable by any party — no pre-registration, no shared secrets, no dependency on a particular server.

## Agents Are Different

Traditional software knows at build time what services it will call and what permissions it needs. Registration, key provisioning, and scope configuration happen before the first request.

Agents don't work this way. They discover resources at runtime. They execute long-running tasks that span multiple services across trust domains. They need to explain what they're doing and why. They need authorization decisions mid-task, long after the user set them in motion. They may need to ask the user questions — or have the user ask them questions — before authorization can proceed.

## How AAuth Addresses These Gaps

| Gap | AAuth Solution |
|-----|---------------|
| No browser | Polling-based token delivery (`202 Accepted` + `Location`) |
| No pre-registration | Agent identity via HTTPS URLs with self-published metadata |
| No independent client identity | Per-instance `aauth:local@domain` identifiers with signing keys |
| Binary auth | Progressive trust: pseudonym → identity → interaction → approval |
| Opaque scopes | Justification parameter + clarification chat during consent |
| Bearer tokens | HTTP Message Signatures on every request, proof-of-possession only |
| Single auth server | Cross-domain PS-to-AS federation |
| No tool governance | Person Server manages tool permissions and audit directly |
| Separate authN/authZ | Unified auth token carries identity and authorization |

## What AAuth Does Not Do

- Does not grant automatic access — services decide what to do with a verified identity
- Does not replace service authorization decisions
- Does not require centralized identity providers
- Complements OAuth for browser-based user login; does not replace it
